# GUI Event Audit

GUI のイベント表面を **全件列挙** し、各イベントが起動するプログラムを **層をまたいで末端まで追跡** して、
「ボタンのラベル／ユーザーの素直な想定」と「実際の挙動」のズレ＝矛盾を発見する。
特に *押しても効かない / キャンセルできない / イベントが重なって制御できない* を、コードを読むだけ（静的解析）で先回りで可視化する。

## Quick Start

1. 人間が監査対象の GUI（リポジトリ／アプリ／画面群）を指定する。
2. **イベント経路の地図**を実例1本で通す（UI宣言層 → ハンドラ層 → サービス層 → IPC/RPC層 → バックエンド実行層 → 結果が戻る道）。
3. **Phase 1**: 全イベント（onClicked / signal / Timer / 購読ハンドラ等）を grep で洗い、**在庫表**にする。
4. **Phase 2**: 優先イベント（cancel/stop・start/実行・破壊的/不可逆操作）を地図に沿って末端まで追跡し、`references/event-audit-lenses.md` の **7レンズ (G1〜G7)** を当てる。
5. 在庫表・発見表・カバレッジ表の **3表** で報告する。

## Workflow

### 0. 対象の特定とイベント経路の地図づくり

- 監査対象の範囲を確定する（アプリ全体か、画面群か）。`grep` / `find` で UI 定義ファイルと、ハンドラ／バックエンドの実体ファイルを特定する。
- GUI イベントは多くの場合 **複数層を貫く**。最初に **1本だけ** 実例を末端まで通し、「どこを読めば実効に辿り着くか」の地図を作る:

  ```
  UI宣言層 (button/onClicked)
    → ハンドラ / ViewModel 層 (@Slot, handler)
      → サービス / ゲートウェイ層
        → IPC / RPC 層 (service client / fetch / IPC)
          → バックエンド実行層 (controller / action / API)
            → 結果が戻る道 (callback / signal / response) → UI の反応
  ```

- 地図の各ホップの **ファイル名と入口関数** を控える。以降のイベントはこの地図に沿って辿る。

### 1. Phase 1 — イベントの全列挙（安価・網羅優先）

UI 定義を走査し、**全イベント源** を1件も漏らさず列挙する。検出対象の例:

- クリック／タップ: `onClicked`, `onClick`, `MouseArea`, `TapHandler`, `addEventListener`, `@click`
- シグナル接続: `Connections { onXxx }`, `signal`, `connect(...)`, `on<EventName>`
- 入力・変更: `onTextChanged`, `onValueChanged`, `onChange`, `onSubmit`
- 時間駆動: `Timer`, `setInterval`, `setTimeout`, 周期購読
- 購読／外部駆動: トピック購読ハンドラ、push 通知、状態変化ハンドラ

各イベントを **在庫表** にする（深追跡はまだしない。一行で「何をすると主張しているか」を書く）:

```markdown
| ID | 画面 / コンポーネント | トリガ | ラベル / 意図 | 呼ぶハンドラ・メソッド | 実効到達? | 場所 |
| --- | --- | --- | --- | --- | --- | --- |
| E1 | CleaningModal | onClicked | 排出を停止 | vm.stop_ejection() | ⚠️ 要確認 | `.../CleaningModal.qml:88` |
```

- 実効到達? の暫定マーク: ✅到達見込み / ⚠️要確認 / ❌空・mock・TODO
- まず全件を埋める。読めなかった箇所は空欄にせず「未取得」と書く。

### 2. Phase 2 — 層またぎ追跡と7レンズの適用（深い）

在庫から **優先イベント** を選び、地図に沿って末端まで追跡する。優先順位の目安:

1. cancel / stop / abort / 中止（偽キャンセルが事故に直結）
2. start / 実行 / 確定 / 送信（楽観的UI・二重発火）
3. 破壊的・不可逆操作（削除・排出・決済・物理動作）
4. 同じ共有状態を触る複数イベント（競合）

各イベントに対し `references/event-audit-lenses.md` の **7レンズ** を当てる:

| 記号 | レンズ | 検出する矛盾 |
| --- | --- | --- |
| G1 | No-op / 死んだ操作 | ボタンは在るが実効なし（mock / TODO / 早期return / 未配線signal / 握り潰し） |
| G2 | 偽キャンセル / 停止 | cancel が REJECT・no-op・False なのに UI は中止扱い／効く窓が狭い |
| G3 | 楽観的UI（結果無視） | 実結果（accepted / 例外）を待たず・見ずに成功・次画面へ遷移 |
| G4 | イベント衝突 / 二重発火 | 多重起動（disable/debounce/in-flight ガード無）、再入同時実行、共有状態競合 |
| G5 | ブロッキング / 無限待ち | ハンドラが UI/executor を長時間・無限占有 → 固まる → 連打誘発 |
| G6 | GUI↔バックエンド 状態乖離 | mode 変更が in-flight 操作に無視、閉じたモーダルへ遅延 signal、store 不整合 |
| G7 | 取りこぼし / 誤配線 | emit したが listen 無し、別画面で発火、ラベルと呼ぶメソッド／遷移先の不一致 |

**追跡の要点**: 行き道（押下→実効）だけでなく **帰り道（結果→UIの反応）** も必ず辿る。末端が「ログだけ / 即return / mock / REJECT / False戻り」で終わっていないか、UI がその実結果に依存して遷移しているかを見る。深い観点が要るときは `new-innovations-pitfall-finder` の F（並行性）/ Q（UX）/ D（例外）観点を併用する。

### 3. 報告（3表）

#### ① イベント在庫表（Phase 1 の全件）

上記の在庫表をそのまま完成形で載せる。

#### ② 発見表（Phase 2 の矛盾）

```markdown
| # | イベント | レンズ | 症状 | 実装の挙動 | ラベル / ユーザーの想定 | 証拠 | 重大度 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 1 | E1 停止ボタン | G2 | 停止が効かない | 下位 action が cancel 未対応で False、排出継続 | 「停止」で必ず止まる | `ctrl.py:260` / `Modal.qml:88` | 🔴高 |
```

該当コードは数行抜粋して引用する。証拠は **行き道と帰り道の両方** の file:line を挙げる。

#### ③ カバレッジ表（何を追跡し、何を未踏で残したか）

```markdown
| イベント / 経路 | 状態 | メモ |
| --- | --- | --- |
| E1 停止 行き道 | ✅ 末端まで | — |
| E1 停止 帰り道 | ✅ 確認 | 結果を無視 → ②-1 で報告 |
| E7 外部API 呼び出し | ⏭ ブラックボックス | 別サービス、未踏 |
```

#### 重大度の目安

| 記号 | 意味 |
| --- | --- |
| 🔴高 | 操作不能・誤操作で事故、データ破壊・物理危険・金銭影響、止まらない停止 |
| 🟠中 | 想定外挙動でユーザー混乱、二重実行、固まりやすい |
| 🟡低 | 配線の冗長・ラベル改善で済む |

## Design Rules

- **網羅優先**: Phase 1 は全イベントを列挙する。深追跡できなかったイベントは「未踏」として在庫表・カバレッジ表に必ず残す。読まずに「問題なし」と書かない。
- **末端＝実効まで読む**: ボタンから物理動作 / DB 書込 / 外部呼び出しなどの **実際の副作用** に届くまで層を辿る。mock / TODO / NotImplemented / REJECT / False 戻りで終わっていないかを必ず確認する。
- **帰り道を必ず辿る**: 結果（callback / signal / response）と、それに対する UI の反応（遷移・表示）まで見る。楽観的UI（結果無視）は帰り道を読まないと見抜けない。
- **ラベルと実効を突き合わせる**: 「ユーザーの素直な想定」はボタンの文言・アイコン・画面遷移から推定し、根拠を1行添える。
- **競合を仮想する**: 同一イベントの連打、別イベントとの重なり、ブロッキング中の操作、閉じたモーダルへの遅延 signal を能動的に想定して当てる。
- **コードベースが正解**: 推測や記憶で挙動を断定しない。必ず該当行を読み、引用する。
- **静的解析の限界を明示**: コードだけでは確定できない挙動は「要実機確認」と明記し、断定しない。
- **重大度を付ける / 対応は強制しない**: 全発見に 🔴/🟠/🟡 を付ける。修正案は示すが設計判断は人間に委ねる。
- **見つからないこともある**: 矛盾 0 件なら 0 件で報告する。無理に絞り出さない。
