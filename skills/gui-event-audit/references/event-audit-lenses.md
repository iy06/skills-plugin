# GUI Event Audit Lenses

`gui-event-audit` の観点カタログと、層をまたぐ追跡の手順。
SKILL.md 本体から呼ばれる詳細リファレンス。

各レンズは次の構造で書く:
- **症状**: コードがどう書かれているか
- **想定とのズレ**: ボタンのラベル／ユーザーが無意識に思い込む挙動
- **検出ヒント**: 何を grep / 何を見れば気づけるか
- **層またぎで見る点**: UI層だけ・バックエンド層だけでは見えず、突き合わせて初めて分かる点

---

## 層をまたぐ追跡のやり方

GUI イベントの矛盾は、UI だけ・バックエンドだけを読んでも見えない。**1イベントを行き道と帰り道の両方で末端まで通す**。

1. **地図を1本作る**: 代表イベントを `UI宣言 → ハンドラ → サービス → IPC/RPC → バックエンド実行 → 結果が戻る道 → UIの反応` で末端まで通し、各ホップのファイルと入口関数を控える。
2. **行き道を辿る**: 押下から **実際の副作用**（物理動作 / DB 書込 / 外部呼び出し）に届くまで追う。末端が `ログだけ` / `即return` / `mock` / `TODO` / `NotImplemented` / `REJECT` / `return False` で終わっていないかを見る（→ G1, G2）。
3. **帰り道を辿る**: 結果（callback / signal / response / 例外）が UI にどう伝わり、UI がどう遷移・表示するかを見る。UI が実結果を **待っているか・分岐しているか** を確認する（→ G3, G6）。
4. **競合を仮想する**: そのイベントを連打したら / 別イベントが重なったら / ブロッキング中に操作したら / モーダルを閉じた後に結果が来たら、どうなるかを当てる（→ G4, G5, G6）。
5. **未踏を明示する**: 別サービス・外部 API などブラックボックスは「未踏」として残す。読めていない箇所を「問題なし」と書かない。

---

## G1. No-op / 死んだ操作

ボタンは在るが、押しても実効が無い。

### G1-1. ハンドラが空 / 即 return

- 症状: `onClicked` の中身が空、`return` のみ、コメントだけ。
- 想定とのズレ: 「押せば対応する処理が走る」。
- 検出ヒント: ハンドラ本体の行数、空ブロック、`pass`、`// TODO` のみ。

### G1-2. mock / 未実装を呼ぶ

- 症状: 呼び先が `mock` / `stub` / `NotImplemented` / 固定値を返すだけ。
- 想定とのズレ: 「本番の処理が走る」。
- 検出ヒント: `mock`, `stub`, `dummy`, `fake`, `NotImplemented`, `raise NotImplementedError`、固定 return。

### G1-3. emit したシグナルに listen が無い

- 症状: `signal`/`emit` するが、対応する `Connections`/listener が存在しない、または名前が不一致。
- 想定とのズレ: 「発火すれば誰かが受け取る」。
- 検出ヒント: emit したシグナル名で listener を grep、シグナル名と handler 名の対応表。
- 層またぎで見る点: 送信側（UI/VM）と受信側（別コンポーネント/別層）を突き合わせないと、片側だけ見ると正常に見える。

### G1-4. 例外を握り潰して結局何もしない

- 症状: `try { 実処理 } catch { /* log のみ */ }` で、失敗時に UI は成功のように振る舞う。
- 想定とのズレ: 「失敗なら気づける」。
- 検出ヒント: 空 catch、log のみ catch、`except: pass`。

### G1-5. disable 条件の論理ミス

- 症状: `enabled` が常時 false（押せない）／常時 true（押してはいけない時も押せる）。
- 想定とのズレ: 「押せる時だけ押せる」。
- 検出ヒント: `enabled`/`disabled`/`is_enabled` のバインディング式、定数化された条件。

## G2. 偽キャンセル / 停止

cancel / stop / abort が、実際には止めない・取り消さない。**最優先で監査する。**

### G2-1. 下位が cancel 未対応（REJECT / no-op）

- 症状: cancel を下位（action / API / デバイス）に伝えるが、下位が未対応で REJECT、no-op、または `return False`。
- 想定とのズレ: 「キャンセルすれば必ず止まる」。
- 検出ヒント: `cancel`/`abort`/`stop`/`中止`/`キャンセル` のハンドラ → 実体、cancel の戻り値、action の cancel callback 実装有無、`goals_canceling` 等の受理判定。
- 層またぎで見る点: UI は「中止」ボタンを出すが、バックエンドの cancel が実装されていない、というギャップは両層を突き合わせて初めて分かる。

### G2-2. 効く窓が狭い（特定状態でのみ受理）

- 症状: cancel が特定状態（待機中・intake 中など）でしか受理されず、進行中（移送・実行中）は無視。
- 想定とのズレ: 「いつ押しても中止できる」。
- 検出ヒント: cancel ハンドラ内の状態ガード（`if state == ...` / `if phase in (...)`）、受理条件の分岐。

### G2-3. UI が cancel の実結果を見ない

- 症状: cancel ボタン押下で **即** 中止画面・元画面へ遷移し、cancel が成功したかを確認しない。
- 想定とのズレ: 「中止表示＝実際に中止された」。
- 検出ヒント: cancel ハンドラ直後の `push`/`navigate`/`close`、結果（False/例外）を分岐していないか。
- 層またぎで見る点: G2-1/G2-2 と組み合わさると「止まっていないのに UI は中止完了」になる。

## G3. 楽観的 UI（結果無視）

非同期処理の結果を待たず・見ずに、UI が成功・次へ進む。

### G3-1. 結果を待たず遷移

- 症状: 呼び出し直後に成功画面・次ステップへ `push`/`navigate`。`await` 忘れ。
- 想定とのズレ: 「成功表示＝実際に成功」。
- 検出ヒント: 呼び出しと遷移が同一ハンドラに並ぶ、`await` の有無、`then` のみで `catch` 無し。

### G3-2. 結果コールバックが分岐していない

- 症状: 完了シグナル/コールバックを受けるが `accepted`/成功フラグで分岐せず、常に成功扱い。
- 想定とのズレ: 「失敗時は失敗画面が出る」。
- 検出ヒント: 結果ハンドラ内の `if accepted` 分岐の有無、失敗パスの遷移先、エラー画面コンポーネントが定義だけされて未到達。
- 層またぎで見る点: バックエンドは正しく失敗（False）を返しているのに、UI 側で読んでいないケースは帰り道を辿らないと見えない。

## G4. イベント衝突 / 二重発火

同じ操作の多重起動、別イベントとの重なり、共有状態の競合。**「別イベントが重なって制御できない」の本丸。**

### G4-1. 多重起動ガードが無い（連打）

- 症状: 押下中に disable / loading / debounce / in-flight フラグが無く、連打で同じ操作が複数走る。
- 想定とのズレ: 「1回押した＝1回実行」。
- 検出ヒント: ボタンの disable 制御、loading state、`is_*ing` フラグ、冪等チェック（unique key / in-flight handle）。

### G4-2. 再入可能群での同時実行

- 症状: コールバック群が再入可能（reentrant）で、複数リクエストが同時に同じハンドラ/コントローラを走る。
- 想定とのズレ: 「順番に1つずつ処理」。
- 検出ヒント: callback group / executor の種類（reentrant か mutually-exclusive か）、共有状態への複数同時アクセス、ロックの有無。

### G4-3. 別イベントが同じ共有状態を書く

- 症状: イベント A と B が同じ store / in-flight ハンドル / モードを書き、片方が他方の状態を上書き・破壊。
- 想定とのズレ: 「各操作は独立」。
- 検出ヒント: 共有 store / グローバル状態 / goal handle への書き込み点の一覧、ロック範囲、check-then-act。
- 層またぎで見る点: UI 上は別々の画面/ボタンでも、バックエンドで同じ状態を触る。両層を突き合わせて初めて競合が見える。

---

## G5. ブロッキング / 無限待ち

ハンドラ/サーバ側が長時間・無限に待ち、UI が固まる。連打（G4）を誘発する。

### G5-1. timeout 無しの結果待ち

- 症状: 結果待ちに timeout が無い（`get()` / `.result()` / `join()` / `wait()` が無限、`float("inf")`）。
- 想定とのズレ: 「数秒で返る / 固まらない」。
- 検出ヒント: RPC/action の結果待ち、`timeout=None`、`float("inf")`、無限ループの待機。

### G5-2. UI スレッド / executor 占有

- 症状: 同期待ちが UI スレッドや限られた executor スレッドを占有し、他のイベントも処理されなくなる。
- 想定とのズレ: 「待っている間も他は動く」。
- 検出ヒント: UI スレッド上の同期待ち、executor のスレッド数 vs 長時間処理の数、ブロッキング呼び出しの callback group。

---

## G6. GUI ↔ バックエンド 状態乖離

画面の状態と、実機/サーバの状態がずれる。

### G6-1. 状態変更が in-flight 操作に無視される

- 症状: mode 変更・設定変更を UI が即反映するが、進行中の処理は古い状態のまま完了する。
- 想定とのズレ: 「変えたら即、全体に効く」。
- 検出ヒント: mode/設定の更新点、in-flight 処理がその値を再参照するか、開始時にコピーして以後見ないか。

### G6-2. 閉じたモーダルへの遅延 signal

- 症状: モーダルを閉じた後に結果 signal が届き、誤って遷移・表示する。または `if (!opened) return` ガードが一部にしか無い。
- 想定とのズレ: 「閉じたら以後の結果は無視される」。
- 検出ヒント: 結果ハンドラ先頭の `opened`/`visible`/`is_*ing` ガードの有無と網羅、モーダル close と結果到達の時間差。

### G6-3. 状態の二重保持で不整合

- 症状: GUI 側 store とサーバ側 store が別々に状態を持ち、片方だけ更新されてずれる。
- 想定とのズレ: 「画面の表示＝実機の状態」。
- 検出ヒント: 同じ概念の状態が GUI とサーバの両方にある、同期するイベントの有無、購読開始前に発火したイベントの取りこぼし。

---

## G7. 取りこぼし / 誤配線

配線が意図とずれている。

### G7-1. ラベルと実効の不一致

- 症状: ボタンの文言・アイコンと、実際に呼ぶメソッド／遷移先が食い違う（コピペ改変漏れなど）。
- 想定とのズレ: 「ラベル通りの処理が走る」。
- 検出ヒント: ラベル文言 vs ハンドラが呼ぶメソッド名 / 遷移先画面、似たボタンの並びでのコピペ。

### G7-2. 発火条件 / 発火画面の誤り

- 症状: 別画面が active の時に発火する、想定外のタイミングで発火する Timer / 購読。
- 想定とのズレ: 「この画面でだけ効く」。
- 検出ヒント: Timer/購読の生存期間、画面状態と発火条件、`enabled`/`running` の制御。

## 適用例（抽象化）

実プロジェクトで実際に出たパターンを、固有名を外して一般化したもの。報告の粒度の参考にする。

### 例1: 止まらない「停止」（G2-1 + G2-3）

- UI: 「停止」ボタン押下で即「停止中」画面へ遷移。
- 実装: 停止要求を下位の長時間 action に転送するが、その action は cancel 未対応で要求を REJECT し、停止関数は `False` を返す。UI はその `False` を見ない。
- 結果: 画面は「停止中」だが、処理は最後まで継続する。
- 報告: レンズ G2、症状「停止が効かない」、行き道（cancel 関数 → action REJECT）と帰り道（UI が False を無視）の両 file:line を証拠に、🔴高。

### 例2: 連打で二重発行（G4-1 + G4-2）

- UI: 実行ボタンに disable / in-flight ガードが無い。
- 実装: ハンドラを処理する callback group が再入可能で、冪等チェックも無い。2回押すと2件のリクエストが同時に下位へ送られる。
- 結果: 同じ操作が二重に実行される。
- 報告: レンズ G4、症状「二重実行」、ガード不在の file:line と callback group 種別を証拠に、🟠中〜🔴高（操作の不可逆性による）。

### 例3: 応答が返らず固まる→連打（G5-1 → G4-1）

- 実装: 結果待ちに timeout が無く、下位が無応答だとハンドラが無限に待つ。
- 結果: UI はローディングのまま固まり、ユーザーが連打し、G4 の二重発行を誘発する。
- 報告: レンズ G5（根本）と G4（誘発される二次被害）を関連付けて報告、🔴高。

### 例4: いつでも押せるが効かないキャンセル（G2-2 + G2-3）

- UI: 「キャンセル」ボタンが常時押下可能。
- 実装: cancel は待機中のみ受理し、進行中は無視。UI は受理結果を見ずに元画面へ戻る。
- 結果: 進行中に押すと、画面は戻るが処理は続く。
- 報告: レンズ G2、受理条件の状態ガードの file:line と UI の即遷移を証拠に、🟠中〜🔴高。
