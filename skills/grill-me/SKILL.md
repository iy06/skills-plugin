# grill-me

ユーザーの曖昧な実装要望に対して、Claude が設計の論点を洗い出し、依存関係の上流から順に質問を投げかけることで、実装前に設計書として合意を取るためのスキル。

## このスキルの目的

AI による実装は速い反面、人間の意図と異なるアウトプットが出ることがある。本スキルは、実装に入る前に Q&A を通じて人間と AI の認識を揃え、合意済みの設計書という形に落とし込むためのもの。

ゴールは「grill されることで、ユーザー自身も自分の要件を言語化できる」状態を作ること。したがって、Claude は受動的に聞くだけでなく、各質問に**推奨回答**を添えて思考の出発点を提供する。

## 全体フロー

1. **実装要望の確認**（要望が呼び出し時に添えられていれば飛ばす）
2. **コードベースの調査**
3. **論点の洗い出しと依存関係の整理**
4. **上流から順に質問**（各質問に推奨回答を添える）
5. **大きな論点が決まるごとに、設計書の該当セクションを提示して合意**
6. **出し切ったらサマリを出し、追加論点をユーザーに促す**
7. **最終的な設計書をまとめてチャット内に提示**
8. **「このまま実装に入りますか？」と確認**し、Yes なら実装へ、保存希望があればパスを聞いて保存

## Step 1: 実装要望の確認

`/grill-me` だけで呼ばれた場合、まず「今回は何を実装したいですか？」と確認する。

`/grill-me ユーザー認証に JWT を追加したい` のように要望が添えられていれば、このステップは飛ばす。

## Step 2: コードベースの調査

質問を始める前に、コードベースをしっかり調査する。コードを読めば分かることは質問しない、というのが本スキルの重要な原則。

### 調査範囲の目安

- 影響しそうなディレクトリのファイル構成
- 既存の類似機能（実装パターン、設計パターン）
- 使われているライブラリ・フレームワーク・バージョン
- コーディング規約（README / CONTRIBUTING / linter 設定 / 既存コードから読み取れるもの）

テスト構成や CI、依存関係の詳細は、設計書を書くのに必要な範囲で踏み込む。タスクの規模や不確実性に応じて調査を深掘りするか判断する。

### 調査後にやること

調査結果から「分かっていること」と「ユーザーに聞かないと分からないこと」を仕分ける。質問は後者だけに絞る。

## Step 3: 論点の洗い出しと依存関係の整理

質問を始める前に、頭の中で設計の論点ツリーを作る。論点間の依存関係を意識する。

- 「A を決めないと B が決められない」という上流・下流の関係を特定
- 上流の論点から順に質問していく
- 後戻りが発生したら、その旨を明示する（例: 「先ほどの X の決定を踏まえると、Y について再検討が必要です」）

論点ツリーを毎回ユーザーに見せる必要はないが、Claude 内部では常に意識する。

## Step 4: 質問の仕方

### 基本ルール

- **質問は一度に一つ**が原則。ただし、関連性の高い複数の論点はまとめて聞いてよい
- **各質問には推奨回答を添える**。理由も短く説明する（なぜその推奨なのか）
- **選択肢形式（A/B/C）で提示する**と、ユーザーが答えやすい
- **推奨回答の根拠はコードベースの調査結果や一般的なベストプラクティス**に基づくものにする

### トーン

丁寧だが徹底的。重要な論点は、ユーザーが流そうとしても粘る。ただし、設計への影響が小さい枝では推奨回答で流してよい。「grill」の本質は設計影響度に応じた徹底さであって、全てを等しく厳しく問うことではない。

### 大きな論点が決まったら

その決定を反映した設計書の該当セクション（ドラフト）を簡潔に見せ、「この方向性で合っていますか？」と確認する。長い設計書全体ではなく、今決まった部分だけをスナップショットで見せるイメージ。

### ユーザーが「分からない」と答えたとき

判断材料が不足している状態。代案を 2〜3 個提示し、それぞれのトレードオフを簡潔に説明して選んでもらう。

### ユーザーが推奨回答に反対したとき

「なぜそうお考えですか？」と意図を軽くヒアリングする。深追いはしない（本題から逸れる）。意図を理解したら、以降の質問もその文脈を踏まえて組み立て直す。

## Step 5: Q&A の終了

Claude が「もう出し切った、これで実装に入れる」と感じたら、ここまでの決定事項をサマリとして提示し、ユーザーに問う:

> 「ここまでの論点を踏まえ、設計書としてまとめる準備ができました。他に詰めておきたい点はありますか？」

ユーザーが追加論点を出せばそれに答え、なければ Step 6 へ。

## Step 6: 設計書の提示

以下のテンプレートに沿って、チャット内に設計書を提示する。該当しないセクションは省略してよい。

```markdown
# <機能名> 設計書

## 要件
### 機能要件
### 非機能要件

## 設計方針

## 変更対象
（ファイル・モジュール一覧、前提条件、スコープ外を含む）

## データモデル / API / 型定義
（該当する場合のみ）

## 処理フロー
（エラーハンドリングを含む）

## テスト方針

## 備考
（未確定事項、今後の検討事項）
```

設計書が小さな変更の場合、要件と変更対象だけで完結することもある。テンプレートに縛られず、タスクに合ったボリュームにする。

### 図の記法：インライン HTML + CSS

「処理フロー」など、構造や流れを図で示す場合は **インライン HTML + CSS** で提示する（Mermaid 記法は使わない）。`<div>` / `<table>` をインラインスタイルで組み、ノードを枠線付きボックス、つながりを矢印（`→` や罫線）で表現する。失敗・分岐などは `background` / `border` / `color` のインライン指定で区別する。外部 CSS / JS ライブラリや `<script>` に依存させず、インラインスタイルだけで自己完結させ、HTML はそのまま描画されるよう応答に直接出力する。

```html
<div style="display:flex;gap:16px;align-items:center;flex-wrap:wrap;font-family:sans-serif">
  <div style="border:2px solid #333;border-radius:8px;padding:10px">入力チェック</div>
  <div>→</div>
  <div style="border:2px solid #333;border-radius:8px;padding:10px">本処理</div>
  <div>→</div>
  <div style="background:#fdd;border:2px solid #c00;border-radius:8px;padding:10px">エラー時のフォールバック</div>
</div>
```

## Step 7: 実装への確認

設計書を提示したあと、次の一言で確認する:

> 「このまま実装に入りますか？」

- **Yes** → 設計書に沿って実装に入る
- **一旦保存したい** → 保存パスをユーザーに聞いて Markdown で保存してから、次の指示を待つ
- **修正したい** → 修正箇所を聞き、必要なら再度 grill して設計書を更新する

保存先のパス指定がない限り、デフォルトではファイル化しない。

## 重要な原則

### コードを読めば分かることは質問しない

これを破ると本スキルの価値が失われる。「このプロジェクトで使われている DB は何ですか？」のような質問は、`package.json` や `Gemfile` や設定ファイルを見れば分かる。調査してから聞く。

### 推奨回答を必ず添える

「どうしますか？」という丸投げの質問はしない。Claude が調査と一般的なベストプラクティスから導いた推奨を、理由とともに添える。ユーザーが同意すればそのまま進み、反対すればなぜかを聞く。これによりユーザーの負担が下がり、対話のテンポが保たれる。

### 依存関係の上流から聞く

下流の詳細（例: エンドポイントのパス）を先に決めてしまうと、上流の決定（例: REST か GraphQL か）が変わったときに無駄になる。論点ツリーを意識し、他の決定に影響する論点を先に片付ける。

### 完璧を目指さない

全ての不確実性を潰そうとすると Q&A が永遠に終わらない。「実装に入れるだけの解像度」が得られたら、残る論点は設計書の「備考」に正直に書き残し、実装フェーズで判断する方が健全。

### grill するのは設計であってユーザーではない

厳しく問い詰めるのではなく、設計を徹底的に詰める。ユーザーが疲れないよう、推奨回答と選択肢形式を使い、対話を軽やかに保つ。

## 下流エージェント (任意)

設計書がまとまったら、`design-review` サブエージェントを呼んで中立な観点でチェックしてもらえる。grill-me の最後に出した設計書が「Claude 自身が grill した成果物」である以上、第三者視点での検証は有効。

- 設計書をファイル保存した場合: パスを渡して呼ぶ (例: 「`docs/design/auth.md` を design-review で見て」)
- チャット内に留めた場合: そのまま貼って呼ぶ

実装が一段落したら、`code-review` サブエージェントで設計との整合性 + コード品質をセルフチェックできる (Bash 権限あり、lint/test も実行可能)。`grill-me → 実装 → code-review` の流れを 1 サイクルとして使うと、PR を出す前の自己検証として機能する。
