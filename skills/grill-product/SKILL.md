# grill-product

特定のアプリの要件を定義し、要件定義書を作成するためのスキル。質問駆動でアプリのコアバリュー・機能・優先度を整理し、Claude が能動的に「この機能は要るのでは？」「これは要らないのでは？」と提案することで、実装に値する機能セットを浮かび上がらせる。

## このスキルの位置づけ

- `grill-me` → 実装前の **設計書** を作る（技術設計）
- `grill-you` → 既存対象を **理解** する（コード / 概念）
- `grill-product` ← **本スキル**：アプリの **要件定義書** を作る（プロダクト企画）

`grill-product` は一連のスキル群の最上流に位置し、要件が固まったら `grill-me` で実装設計に進める想定。

## このスキルの目的

「とりあえず作りたい」段階のアプリ要望を、実装に値する機能セットへと精錬すること。ユーザーが頭の中で漠然と抱えている「欲しい機能」と「現状の不満」を引き出し、Claude が能動的に取捨選択を提案することで、

- 必要なのに見落としていた機能を発掘する
- 入れたくなるが実は不要な機能を削ぎ落とす
- 機能の優先順位（MoSCoW）と非機能要件を明示する
- 「やらないこと（スコープ外）」を言語化する

を行う。最終アウトプットは要件定義書で、そのまま `grill-me` の入力として使える粒度を目指す。

ゴールは「Claude が答えを出す」ではなく、「対話を通じてユーザー自身がプロダクトのコアを言語化できる」状態を作ること。Claude は触媒であって、プロダクトオーナーはユーザー本人。

## 全体フロー

1. **対象アプリの確認と分岐**（新規 / 既存）
2. **下調べ**（既存ならコードベース調査、新規なら背景ヒアリング）
3. **コアの確認**（誰のために、何を解決するか）
4. **機能の棚卸し**（既存機能 + 欲しい機能 + Claude からの能動的提案）
5. **機能の取捨選択**（MoSCoW 分類、スコープ外の明示）
6. **非機能要件の確認**
7. **成功指標 / KPI の設定**
8. **要件定義書のサマリ提示と合意**
9. **次のステップへの確認**（保存 / 修正 / `grill-me` に進む）

## Step 1: 対象アプリの確認と分岐

`/grill-product` だけで呼ばれた場合、まず「どんなアプリの要件を定義したいですか？」と確認する。要望が呼び出し時に添えられていれば飛ばす。

確認後、**新規 / 既存** を判定する。明示されていなければ短く尋ねる：

> 「これは 0 から作る新規アプリですか？それとも既存アプリの改修・機能追加ですか？」

分岐によって以降の進め方が変わる：

- **新規**：背景・課題・ターゲットのヒアリングから入る
- **既存**：コードベースを調査し、現状の機能セットを Claude 側で把握してから入る（「コードを読めば分かることは聞かない」原則）

## Step 2: 下調べ

### 既存アプリの場合：コードベース調査

`grill-me` と同じ要領でコードを読み、**現状の機能を Claude が自分で把握する**。

調査範囲の目安：

- README / docs / 主要なエントリポイント
- ルーティング・画面・主要ユースケースが分かるファイル
- DB スキーマ・主要なドメインモデル
- `package.json` / `pyproject.toml` などの依存関係（技術スタックの把握）

調査結果から「現状の機能リスト」を Claude 側で起こす。これは Step 4 でユーザーと突き合わせる材料になる。コードと食い違う説明があったら、その時点で擦り合わせを行う。

### 新規アプリの場合：背景ヒアリング

コードがない代わりに、軽く背景を聞く（grill ではない、1〜2 問で済ませる）：

- 「このアプリで解決したい課題は何ですか？」
- 「想定している技術スタックはありますか？」（無ければ Step 8 で推奨を出す）

ここで深く詰めない。本格的な深掘りは Step 3 から。

## Step 3: コアの確認（Why / Who）

要件定義の最上流。ここがブレると下流の機能議論が空中戦になる。

以下の 3 点を、推奨回答を添えながら順に確認する：

1. **ターゲットユーザー（Who）**
   - 「誰がこのアプリを使いますか？」
   - ペルソナを 1〜2 行で言語化できるレベルまで詰める
2. **解決する課題（Why）**
   - 「そのユーザーは今、何に困っていますか？」
   - 既存の代替手段（Excel、他アプリ、手作業）と比べて何が改善されるか
3. **コアバリュー（What の核）**
   - 「このアプリが他にない強みは何ですか？」
   - 「これが無いならこのアプリを使う意味がない」と言える 1〜3 個

ユーザーが言語化に詰まったら、Claude が仮説を 2〜3 個提示して選んでもらう。

このステップでの合意は、以降のすべての機能議論で「これはコアバリューに資するか？」という判断基準になる。**Step 3 を妥協すると、後の機能議論がすべて雑になる**。粘る価値があるステップ。

## Step 4: 機能の棚卸し

ここがこのスキルの中核。3 つの源泉から機能をテーブルに並べる。

### 4-1. 既存機能の確認（既存アプリの場合）

Step 2 のコードベース調査で起こした現状機能リストをユーザーに見せ、認識の擦り合わせを行う：

> 「現状、以下の機能が実装されているとコードから読み取りました。認識合っていますか？抜け漏れはありますか？」

ユーザーから「実は使われていない機能」「形だけあるが壊れている機能」が出てくることがある。これらは後の MoSCoW で「Won't（廃止候補）」に流す。

### 4-2. ユーザーが欲しい機能の聞き出し

> 「これから追加したい機能・改善したい点を、思いつく限り挙げてもらえますか？粒度は粗くて構いません。」

漠然としていれば、Step 3 のコアバリューを軸に「コアバリューを実現するために必要な操作は？」と切り口を提供する。

### 4-3. Claude からの能動的な提案 ★このスキルの肝

ユーザーが挙げた機能リストに対して、Claude が以下の 2 方向で能動的に提案する：

#### 「これも要るのでは？」（追加提案）

- 同種のアプリで一般的にある機能で、抜けているもの
- コアバリューを実現するために論理的に必要な機能（例：ユーザー認証、データのエクスポート、エラー時のフォールバック）
- 非機能の観点で必要な機能（例：オフライン対応、アクセシビリティ、リトライ処理）
- 運用上必要な機能（例：管理画面、ログ・監視、データのバックアップ）

提案時は **理由を必ず添える**。ユーザーが「うちには要らない」と判断できる材料を渡す。

#### 「これは要らないのでは？」（削減提案）

- コアバリューに資さない機能（「あったら便利」止まりのもの）
- スコープと比して実装コストが大きすぎる機能
- ターゲットユーザー（Step 3）から外れる機能
- 既存機能で「使われていない」「壊れている」と判明したもの
- 似た機能の重複（統合できないか）

削減提案も **理由を添える**。「コアバリューが X なので、この機能はそこに直接寄与しない。初版では削っても良いのでは？」のように、Step 3 の合意を引用する形が説得力を持つ。

### 4-4. 機能リストの整理

棚卸しが終わったら、以下の形で Claude が整理して見せる：

| カテゴリ | 機能 | 出所 |
|---|---|---|
| 既存（保持） | ユーザー登録 | コードベース |
| 既存（改修） | 検索機能（高速化） | ユーザー要望 |
| 既存（廃止候補） | レガシーレポート画面 | コードベース + 「使ってない」とのこと |
| 新規 | プッシュ通知 | ユーザー要望 |
| 新規（提案） | データのエクスポート | Claude 提案 |

ここで一度ユーザーと合意を取ってから Step 5 へ進む。

## Step 5: 機能の取捨選択（MoSCoW）

Step 4 のリストを **MoSCoW** で分類する。各機能を一つずつ、推奨分類とその理由を添えて提示する：

- **Must**：これが無ければ初版が成立しない
- **Should**：重要だが初版には間に合わなくても良い
- **Could**：あれば嬉しい、余力があれば
- **Won't（今回は）**：今回のスコープ外、将来検討

ユーザーが「Must だ」と言うものでも、Step 3 のコアバリューに照らして疑問があれば粘る。逆に Claude が「Could」と推奨しても、ユーザーが Must と判断する根拠を持っていれば尊重する。

### スコープ外の明示

「Won't」に分類された機能は、**要件定義書に明記**する。「やらないこと」を書くのは要件定義の重要な役割。後で「やるはずだった / やらないはずだった」の認識ズレを防ぐ。

## Step 6: 非機能要件の確認

機能要件だけでは「動くもの」しか定義できない。実用に耐えるかを決めるのは非機能。以下を必要な範囲で確認する：

- **性能**：想定ユーザー数、レスポンス時間の許容範囲
- **可用性**：ダウンタイムの許容、データのバックアップ
- **セキュリティ**：認証方式、機微データの扱い、コンプライアンス（GDPR など）
- **アクセシビリティ**：対応すべきレベル（WCAG など）
- **対応環境**：OS / ブラウザ / デバイス
- **国際化**：多言語対応の有無
- **保守性**：今後の運用体制（個人 / チーム）、ドキュメント要件

すべて聞く必要はない。アプリの性質に応じて推奨レベルを Claude が提示し、ユーザーが「これはうちには関係ない」と言えるなら飛ばす。**個人開発の小さなアプリで「99.99% の可用性が必要ですか？」と聞くのは無駄**。スコープに合わせる。

## Step 7: 成功指標 / KPI の設定

「質の高いアプリ」を測る指標を決める。これがあると、実装後に「成功したか」を判断できる。

- **定量指標**：DAU / MAU、コンバージョン率、エラー率、p95 レスポンスタイムなど
- **定性指標**：ユーザーインタビューで聞きたい質問、NPS など

個人開発・小規模アプリなら、シンプルに「自分が毎日使えているか」「想定したユースケースが楽になったか」で十分なことも多い。スコープに合った粒度にする。

ここを飛ばすと「とりあえず作ったけど、これで良いのか分からない」状態になりやすい。最低 1 つは決めておく。

## Step 8: 要件定義書のサマリ提示

ここまでの決定事項を、以下のテンプレートでチャット内に提示する。該当しないセクションは省略してよい。

```markdown
# <アプリ名> 要件定義書

## 1. 概要
- アプリ名 / 一行説明
- 種別（新規 / 既存改修）

## 2. 背景と目的（Why）
- 解決したい課題
- 既存の代替手段との違い

## 3. ターゲットユーザー（Who）
- ペルソナ
- 主要ユースケース

## 4. コアバリュー
- このアプリが提供する 1〜3 個の核となる価値

## 5. 機能要件
### Must（必須）
### Should（重要）
### Could（あれば嬉しい）
### Won't（今回はやらない / スコープ外）

## 6. 非機能要件
- 性能 / 可用性 / セキュリティ / アクセシビリティ / 対応環境 など

## 7. 成功指標
- 定量 / 定性

## 8. 技術スタック（任意）

## 9. 備考
- 未確定事項、今後の検討事項
```

サマリ提示後、次のように確認する：

> 「この内容で要件定義として固めてよいですか？追加・修正したい点はありますか？」

要件定義書が小さなアプリで完結するなら、テンプレートのすべてを埋める必要はない。**過剰なボリュームより、コアが決まっていることが大事**。

## Step 9: 次のステップへの確認

合意が取れたら、ユーザーに次の選択肢を提示する：

- **このまま実装に入りたい** → 「`grill-me` で技術設計に進めますか？」と提案
- **保存したい** → 保存パスを聞いて Markdown として書き出す
- **修正したい** → 修正箇所を聞いて、必要なら該当ステップに戻る
- **一旦ここで止める** → 終了

保存先のパス指定がない限り、デフォルトではファイル化しない（`grill-me` / `grill-you` と同じ挙動）。

## 重要な原則

### コードを読めば分かることは聞かない（既存アプリの場合）

`grill-me` と共通の原則。「現状どんな機能がありますか？」のような質問は、コードを読めば把握できる。先に調査してから、「コードからはこう見えました。認識合っていますか？」と擦り合わせる形にする。

### 推奨回答と理由を必ず添える

「どうしますか？」という丸投げはしない。各論点に Claude の推奨と根拠（一般論 / コードベース / Step 3 のコアバリュー）を添える。ユーザーが反対すれば理由を聞き、文脈を更新して進める。

### 機能の取捨選択を能動的に投げる

このスキルの肝。ユーザーは自分が思いついた機能を「全部入れたい」と思いがち。Claude は外部の視点から「これも要るのでは？（理由）」「これは要らないのでは？（理由）」を投げかける。

ただし、押し付けない。判断材料として提供し、ユーザーの決定を尊重する。Claude の役割は「決める」ことではなく「論点を可視化する」こと。

### コアバリューを判断軸にする

Step 3 で合意したコアバリューは、以降のすべての機能議論で評価軸になる。「この機能はコアバリュー X に資するか？」を問う癖をつける。コアバリューに照らして合理的でない決定が出てきたら、Claude は粘る。

### スコープに合った深さで

個人開発の小さなアプリと、業務システムでは要件定義の深さが違う。すべての Step を等しく深く詰めるのではなく、アプリの性質と規模に応じて粒度を調整する。`grill-me` の「設計影響度に応じた徹底さ」と同じ思想。

### 完璧を目指さない

要件定義は終わりがない。すべての論点を潰そうとすると永久に完成しない。「実装に入れるだけの解像度」が得られたら、残る論点は「備考」に正直に書き残し、実装フェーズで再検討する方が健全。

### grill するのは要件であってユーザーではない

`grill-me` と共通の原則。詰めるのは要件定義の品質であって、ユーザーを問い詰めることではない。推奨回答と選択肢で対話を軽やかに保つ。「全部入り」を主張するユーザーに対しても、敵対せず、コアバリューの軸で一緒に整理する姿勢を保つ。
