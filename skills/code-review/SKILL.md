# Code Review Skill

変更されたコードに対して、セキュリティ・アーキテクチャ・パフォーマンス・可読性の観点で徹底的にレビューを行い、問題点と改善案を具体的に指摘する。

## Workflow

### 1. レビュー対象の特定

ユーザーが対象を指定していない場合は以下の優先順位で特定する：

1. ユーザーが明示したファイルやディレクトリ
2. `git diff HEAD` でステージング済みの変更
3. `git diff main...HEAD` でブランチ全体の変更
4. 直近で変更されたファイル

対象が広すぎる場合はユーザーに絞り込みを確認する。

### 2. コードの読み込み

対象ファイルを読み込み、変更の意図・コンテキストを把握する。必要であれば関連ファイル（テスト、型定義、設定など）も確認する。

### 3. 4つの観点でレビュー

以下の観点を**すべて**確認する。問題がない場合も「問題なし」と明記する。

#### 🔒 セキュリティ

- 入力バリデーション・サニタイズの漏れ
- SQLインジェクション / XSS / CSRF などの脆弱性
- 認証・認可の欠陥
- シークレット・APIキーのハードコード
- 依存ライブラリの既知の脆弱性（バージョン指定が古すぎる場合）
- エラーメッセージへの機密情報の露出

#### 🏗️ アーキテクチャ

- 単一責任の原則（SRP）の違反
- 過度な結合・依存関係の問題
- 適切な抽象化がされているか
- 既存パターン・規約との一貫性
- テスタビリティの問題
- 将来の変更に対する柔軟性

#### ⚡ パフォーマンス

- N+1クエリや不必要なループのネスト
- 不必要なメモリ割り当て・コピー
- キャッシュすべき処理のキャッシュ漏れ
- ブロッキング処理・非同期処理の問題
- データベースインデックスの考慮
- 大量データ処理時のスケーラビリティ

#### 📖 可読性

- 命名の明確さ（変数・関数・クラス名）
- 関数・クラスの長さが適切か
- マジックナンバー・マジック文字列の使用
- コメントの適切さ（不必要なコメント・必要なのにないコメント）
- エラーハンドリングの一貫性
- コードの重複（DRY原則の違反）

### 4. レビュー結果の報告

以下のフォーマットで報告する：

```
## コードレビュー結果

### 📋 サマリー
- 対象: <ファイル名・変更範囲>
- 重大な問題: X件 / 軽微な問題: Y件 / 提案: Z件

---

### 🔒 セキュリティ
<問題点があれば重大度付きで列挙。ファイル名と行番号を必ず記載。>
<問題がなければ「✅ 問題なし」>

### 🏗️ アーキテクチャ
<同上>

### ⚡ パフォーマンス
<同上>

### 📖 可読性
<同上>

---

### 🎯 優先対応リスト
重大度順に対応すべき項目をリストアップする。
1. [重大] ...
2. [中] ...
3. [軽微] ...
```

### 5. 修正案の提示

重大な問題には必ず具体的な修正コードを提示する。軽微な問題や提案には改善のヒントを添える。

ユーザーが修正を依頼した場合はそのまま対応する。

## 重大度の定義

| 重大度 | 説明 | 例 |
|---|---|---|
| 🔴 重大 | 本番環境で即座に問題になる / セキュリティリスク | SQLインジェクション、認証バイパス |
| 🟠 中 | 将来問題になる可能性が高い / パフォーマンスへの影響大 | N+1クエリ、メモリリーク |
| 🟡 軽微 | 改善が望ましい / ベストプラクティスから外れている | 命名の改善、コードの整理 |
| 💡 提案 | より良くする余地がある | リファクタリング案、代替アプローチ |
