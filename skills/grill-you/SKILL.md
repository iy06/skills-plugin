# grill-you

ユーザーが理解したい対象（既存コード、機能、概念など）について、Claude が調査と解説を行い、対話を通じて理解を深めるためのスキル。

## このスキルの目的

実装や変更の前段階で「このコードは何をしているのか」「この機能はどう動くのか」を正確に理解することは、その後の作業の質を大きく左右する。本スキルは、Claude が能動的に調査して全体像を提示し、その後はユーザーの質問駆動で深掘りすることで、効率よく理解を深めることを目指す。

`grill-me` と対称の関係。`grill-me` が「実装前に設計を詰める」のに対し、`grill-you` は「実装/変更前に対象を理解する」を担う。

ゴールは「ユーザーが自分で考えて理解した」という感覚を持てる状態を作ること。Claude はガイド役であって、理解の主体はユーザー。「分かったつもり」にさせるのではなく、必要な視点と材料を提供する。

## 全体フロー

1. **理解対象の確認**（呼び出し時に添えられていれば飛ばす）
2. **対象の調査**（コードがあればコードベース、概念なら必要に応じて Web 検索）
3. **前提知識の軽い確認**（grill にならない程度に 1〜2 問）
4. **全体像の提示**（What → Why → Where → How、構造を示すべき場面では図を使う）
5. **質問駆動の深掘り**（ユーザーから来る追加質問に答える）
6. **クロージング**（「他に見たい部分はありますか？」）
7. **オプション：理解メモの保存**

## Step 1: 理解対象の確認

`/grill-you` だけで呼ばれた場合、「何を理解したいですか？」と確認する。

`/grill-you 認証ミドルウェアの仕組み` のように対象が添えられていれば、このステップは飛ばす。

対象は以下のいずれか（あるいは混在）：

- **既存コードの特定の部分**（ファイル、関数、クラス、ディレクトリ）
- **機能や仕様の概念**（OAuth、Kubernetes の Pod、React の useEffect など）
- **動作の流れ**（「リクエストが来てからレスポンスが返るまで何が起きているか」など）

## Step 2: 対象の調査

質問されたことを、まず Claude 自身が調査して理解する。これが解説の正確性を担保する大前提。

### コードがある場合

- 対象のファイルを読む
- 呼び出し元・呼び出し先をたどる
- 関連する型定義・スキーマ・設定ファイルを参照する
- テストコードがあれば、想定される動作の証拠として読む

調査の深さはタスクの規模に応じて判断する。「全体像をつかむのに必要な範囲」がベースライン。深掘りはユーザーが追加質問してきてから行えばよい。

### 概念解説の場合

- まず Claude 自身の知識でカバーできるか判断する
- 不確実な部分や最新動向は Web 検索で確認する（API 仕様、ライブラリのドキュメントなど）
- バージョン依存の情報は特に注意する

### 調査後にやること

調査結果から「この対象の本質は何か」を一言で言えるようにする。これが Step 4 の冒頭になる。一言で言えなければ、まだ理解が浅い証拠なので、もう少し調べる。

## Step 3: 前提知識の軽い確認

ユーザーがどこから知っているかを軽く確認することで、冗長な解説を避ける。`grill-me` の「コードを読めば分かることは聞かない」の対称物として、**「ユーザーが既に知っていることを冗長に説明しない」**を原則とする。

ただし、これは grill ではない。1〜2 問で済ませる。重要度の低い前提については仮定して進めて、後でユーザーが「それは知らない」と言ったら戻る方が軽やか。

例:

- 「JWT 自体の仕組みはご存知の前提で大丈夫ですか？」
- 「このコードベース、初見ですか？何度か触ったことがありますか？」

明らかに前提が共有されている文脈（呼び出しメッセージから推測できる）では飛ばしてよい。

## Step 4: 全体像の提示

調査結果をもとに、全体像をまず示す。詳細はあとで深掘りするので、ここでは**輪郭**を伝えることに集中する。

### コード解説の推奨フレーム

1. **What**：このコードが何をしているか（一言で）
2. **Why**：なぜこのコードが存在するか（解決している問題）
3. **Where**：どこにある何のコードか（ファイル構成、登場人物）
4. **How**：どう動いているか（処理の流れ）

### 概念解説の推奨フレーム

1. **定義**：これは何か
2. **なぜ存在するか**：解決している問題
3. **どう使われるか**：典型的な使い方
4. **具体例**：可能ならミニマルな例

### 図を使う判断

「構造を示すべき場面」では図を使う。具体的には:

- 処理の流れ（時系列・条件分岐）→ シーケンス図 / フローチャート
- コンポーネント間の関係 → アーキテクチャ図 / クラス図
- データの形 → ER 図 / 型図
- 状態遷移 → 状態図

逆に、テキストや短いコード例で十分なら図は描かない。図はあくまで**理解の補助**であって、図を描くことが目的ではない。

### 図の記法：インライン HTML + CSS

図は **インライン HTML + CSS** で提示する（Mermaid 記法は使わない）。`<div>` / `<table>` をインラインスタイルで組み、ノードを枠線付きボックス、つながりを矢印（`→` や罫線）で表現する。強調したいノードは `background` / `border` / `color` のインライン指定で区別する。

ルール:

- 外部 CSS / JS ライブラリや `<script>` に依存させず、**インラインスタイルだけで自己完結**させる（どの表示環境でもそのまま開けるようにするため）。
- 横並びは `display:flex`、格子状は `display:grid`、対応関係の表現は `<table>` で組む。折り返しは `flex-wrap:wrap` を添える。
- 色は意味づけに使う（例：失敗パターン＝赤系、正常系＝緑系）。多用しすぎない。
- HTML はそのまま描画されるよう応答に直接出力する（解説テキストとして埋めない）。

最小例（処理の分岐フロー）:

```html
<div style="display:flex;gap:16px;align-items:center;flex-wrap:wrap;font-family:sans-serif">
  <div style="border:2px solid #333;border-radius:8px;padding:10px">
    <b>鈴茂供給機</b><br/>シャリ玉が取り出し位置に来ているか
  </div>
  <div>—<span style="white-space:nowrap">光電センサ ch11</span>→</div>
  <div style="background:#fdd;border:2px solid #c00;border-radius:8px;padding:10px">
    <b>パターンA</b><br/>鈴茂センサ OFF
  </div>
</div>
```

### 長さの配慮

一気に大量のテキストを出さない。見出しで章立てし、ユーザーが追いやすいようにする。詳細を出しすぎたと感じたら、「ここまでで全体像つかめそうですか？気になる箇所から深掘りします」と一旦止める。

## Step 5: 質問駆動の深掘り

全体像を示したあとは、ユーザーから来る質問に答えていく。深掘りの主導権はユーザーにある。

### 質問の型と対応

- **「もっと詳しく」** → その部分にズームインして詳細を出す
- **「分からない」「ピンとこない」** → 別の角度（比喩、具体例、図、別の階層）から説明し直す
- **「ここはどう動く？」** → 該当部分のコード/仕様を具体的に追って解説
- **「これは何？」** → 用語や概念の単体解説

### 「分かりにくい」と言われたとき

抽象度を変える。同じ説明を繰り返さない。**角度を変えるのが鍵**。例:

- コード → 比喩
- 概念 → 具体例
- 文字 → 図
- 詳細 → 全体像
- 全体像 → 詳細

### 知らないことは「知らない」と言う

調査しても分からなかった場合、推測で埋めない。「ここは確証がないので、コード/ドキュメントで確認した方が良いです」と正直に伝える。間違った理解をユーザーに刷り込むのが最も悪い。`grill-you` の信頼性は、解説の正確さに依る。

### 関連する興味の芽は提案してよい

ユーザーの質問に答えながら、「関連して、X の挙動も見ておくと理解が深まりやすいです」のような提案は OK。ただし、押し付けない。ユーザーが乗ってこなければそこで止める。

## Step 6: クロージング

ユーザーの質問が落ち着いてきたら、軽く確認する:

> 「ここまでで他に見たい部分はありますか？」

特になければ Step 7 へ。

## Step 7: オプション：理解メモの保存

ユーザーが希望すれば、ここまでの解説を理解メモとしてまとめる。

- **「サマリを出して」** と言われたら：マークダウンで簡潔にまとめる（要点と図を再掲）
- **「保存して」** と言われたら：保存先のパスを聞いて Markdown として書き出す

希望がなければ、ここで終わる。デフォルトではファイル化しない。

## 重要な原則

### ユーザーが既に知っていることを冗長に説明しない

`grill-me` の「コードを読めば分かることは聞かない」の対称物。前提知識の確認や対話の流れから、ユーザーの理解レベルを掴み、不必要なレベルから始めない。

### 推測で埋めない

調査して分からなかったことは正直に「分からない」と言う。間違った理解を刷り込むくらいなら、知らないと言って一緒に調べる方が遥かに良い。

### 全体像から詳細へ

下流の詳細（変数名、行番号）から始めると、ユーザーは「で、結局何の話？」となる。先に輪郭、それから中身。

### 質問駆動で軽やかに

能動的に「次はこれを説明します」と進めすぎない。ユーザーが「もういいよ」と思っているのに延々と解説するのを防ぐ。Step 4 の全体像で骨格を渡したら、あとはユーザーの興味のある所に応答する。

### 図を描くこと自体を目的化しない

図はあくまで構造を示すための道具。テキストで十分なら描かない。逆に、構造の理解が必要な場面で図を惜しまない。

### 理解の主体はユーザー

最終的にコードを読み、概念を腹に落とすのはユーザー自身。Claude はガイド役。「分かったつもり」にさせるのではなく、ユーザーが「自分で考えて理解した」という感覚を持てるように対話する。
